# L2 — Не Иннокентий, а Кеша (`L2_KESHA_CACHE`)

**Блок прогрессии:** B (компромиссы)  
**Статус:** `data-in`

---

## Нарратив

**Заказчик:** Иннокентий «Кеша»  
**Предмет:** Шар-ускоритель «Редиска» (кэш-сервис)

**Вступление:**
> Нет, ну ты же понимаешь, что Иннокентий — это официально, а на самом деле ты меня Кешей зови. Тут вот какое дело. Я себе сделал такую штуку — Редиска называется — шар-ускоритель. Все данные, которые часто нужны, он запоминает. И когда нужно что-то найти в архиве, он сразу отдаёт — архив не гоняешь.
>
> Начальство увидело и говорит: «Иннокентий, сделай для всех». Ну, для Кеши, то есть. Теперь каждый отдел хочет: морковка, капуста, картошка. Все сразу.
>
> Проблема — Редиска была для меня одного. Там не было никакой изоляции между отделами. Отдел финансов залил туда все счета квартала — для быстрого доступа. Как бы не случилась суматоха.

**Финал:**
> Ну, Редиска теперь умеет говорить «нет» — справедливо и по квотам. Финансовый отдел больше не захватит всю память. Уважаю.

---

## Технологический смысл

**Что происходит:** Redis-кэш, созданный для одного пользователя, превращается в **Cache-as-a-Service** с поддержкой нескольких арендаторов. Нужно добавить изоляцию между арендаторами и управление квотами.

Инженерный урок:
- Шарить ресурс без изоляции → данные разных арендаторов смешиваются (F3_COMPROMISE).
- Шарить ресурс без квот → один арендатор занимает всё место (F1_OVERFLOW).
- TTL чистит старое, но не ограничивает живых потребителей — для CaaS нужны квоты.

**Новые механики (Block B — вводим впервые):**
- **Опциональные матрицы** (`maxOptionalMatrices=1`) — нужно выбрать одну из двух матриц, каждая открывает разный путь решения.
- **R_ONBOARDING** — новый тип обряда: снижает ауру (не монеты). Противопоставляется R_AI_BUFF и R_LIBRARY_GRANT, которые снижают монеты.

**Закрепляем из L1 (`reuse`):**
- Наследованные долги — Редиска принесла два стартовых долга (D_CACHE_GROWTH, D_OPEN_ACCESS), как замок приносил слепоту.
- Обряды-субсидии — R_LIBRARY_GRANT знаком по механике (монетная субсидия), но теперь нужен в другом контексте.
- R_AI_BUFF — знаком из L1, но здесь аура лимит 3 (а не 5). Использование R_AI_BUFF (+3 ауры) неизбежно приведёт к превышению лимита — это намеренная ловушка на закрепление.

## Заложенный урок / компромисс

**TTL ≠ квоты.** TTL чистит старые записи — полезно, но не ограничивает живых потребителей прямо сейчас. Для CaaS нужны `S_QUOTAS`.

**Два честных пути к изоляции:**
- **Arch (технический):** M_CONNECTIVITY + S_AUTHZ — формальная гарантия, но дороже (6 монет, точно в бюджет). Без R_LIBRARY_GRANT не вмещается. Платишь за надёжность.
- **Org (организационный):** M_OBSERVATION + R_ONBOARDING — дешевле (5 монет, есть запас), убирает ауру. R_ONBOARDING работает только в RITUAL_OBS: нельзя «договориться» без инфраструктуры наблюдения.

**R_AI_BUFF знаком, но контекст изменился.** Игрок уже видел R_AI_BUFF в L1 (там он был обязателен). Здесь auraLimit=3 вместо 5 — R_AI_BUFF даёт +3 ауры и немедленно обрушивает лимит. Один и тот же обряд, другой результат в другом контексте.

**Обряды бывают разные.** R_LIBRARY_GRANT и R_AI_BUFF — оба монетные субсидии. R_ONBOARDING — аура-снижение. Выбор обряда зависит от того, чего именно не хватает.

## Подсказки

1. «Редиска унаследовала два долга — как замок мельника нёс слепоту. Оба нужно закрыть.»
2. «Квоты — единственный способ доказать, что шар умеет справедливо делиться. TTL только чистит старое, но не ограничивает нынешних потребителей.»
3. «Нужна матрица — но только одна. Выбери: Связи дают технический замок, Взгляд открывает путь через договор.»
4. «Путь через Связи стоит на монету больше бюджета — нужна монетная субсидия. Но осторожно: у обрядов разная цена по ауре.»
5. «Печать Прогресса снова в деле — но здесь аура лимит втрое жёстче, чем у мельника.»
